Stage 2: Designing Conceptual Model

In this stage, the Development Team works with the rest of the Working Group in order to iteratively design the Conceptual Model (CM) of the module to be released. The CM is represented as a collection of UML diagrams in an Enterprise Architect file in the OP/epo-conceptual-model repository.

The first business process of this section is “Trunk-based development of the Model”. This is an iterative process that will happen multiple times in this stage. The first time that this process happens, the Development Team will create the first draft of the Conceptual Model as a UML diagram in an Enterprise architect file and store it on the develop branch of the OP/epo-conceptual-model repository. During all the subsequent times that this process takes place, the Development team will continue editing and updating the Conceptual Model in order to implement feedback received in the Working Group Meetings. As a result of this business process, the “Conceptual Model (editable)” business object is created or updated. Also In this Process, the Development Team implements any GitHub issues pointed by the Product Owner in the “Decide work goal & scope and release version” business process or any GitHub issues created during the ”CM Working Group Meeting” business process.

In the “Dev Team internal review” business process, The Development Team uses model2owl on the current Conceptual Model in order to generate a conventions report. See here on how model2owl is used to generate a conventions report. For a conventions report example see here. The Development Team consults the conventions report and the ORSD, and makes adjustments to the Conceptual Model if needed, going to “Trunk-based development of the Model” or “Fixing Model Bugs in the release version” business process. If no changes are currently needed, the next business process will be “CM Working Group Meeting”.

In the ”CM Working Group Meeting” business process, that takes place during the weekly Working Group meeting, the Development Team Presents and explains the conceptual model to the rest of the Working Group. The Working Group validates the model and develops it further. By the end of the meeting, the Development Team either:

  • Receives change requests that may take the form of GitHub issues, going to “Trunk-based development of the Model”

  • Receives the final validation of the model (if development is finished) and can proceed to the next stage.

Changes to the Conceptual Model may be needed as a result of a review from a later stage of the workflow such as the Product Owner reviewing the SDS, in the “Product Owner internal review” business process, or the Wider Public reviewing the new release candidate during the “Review the release candidate” business process.

  • If changes are needed after the “Product Owner internal review” business process, then the changes happen on the develop branch.

  • If the changes happen during the “Review the release candidate” business process by the Wider Public (read more about this business process here), then the “Fixing Model Bugs in the release version” business process is invoked. During this process, the Conceptual Model file is updated in the release branch of the OP/epo-conceptual-model.

In the “Porting the bug fixes into the trunk” business process, the changes done in the previous “Fixing Model Bugs in the release version” business process are replicated in the develop branch of the OP/epo-conceptual-model. This replication happens because there is no possibility to edit EA files in parallel and then have an organic easy merge. However, if no new trunk-based development has happened on the develop branch (e.g., no new modules have been developed), then this can be done as a simple branch merge.

Conceptual Model Conventions

The ePO Conceptual Model is designed according to the SEMIC style guide UML Conceptual model conventions and follows model2owl UML covnetions so that model2owl can be used for SDS artefact generation.

ePO concept life Cycle

Regarding the ePO life cycle, conventions for controlling which concepts of the Conceptual Model can be published/released are introduced in the form of properties for these elements (status labels for each concept in the EA file as depicted in the image below). These conventions take the form of the following states:

  1. Proposed: The concept is under Working Group discussion or has not been discussed yet. In the latter case, the concept is in the “sandbox” subfolder of the CM file.

  2. Approved: The concept has a Working Group approval tag and should be moved to its respective module subfolder.

  3. Implemented: The concept has been published in the current stable ePO release or is ready for publication in the upcoming release.

The three aforementioned states form the ePO concept lifecycle as depicted in the figure below.

Note that due to limitations on EA, Proposed, Approved and Implemented ((pre-)release stages) are only possible for Classes.

Deprecation
A fourth ePO concept state exists, but it is not part of the CM: Deprecated. The concept was published in a previous release and is no longer relevant. Deprecated concepts are not part of the Conceptual Model and only exist as references in a dedicated OWL-core file that is created automatically by model2owl by comparing the most recent version of ePO with the older versions. See stage 3 for more information.

Metadata conventions for Internal and External CM concepts

Here, we mention the different metadata conventions used for ePO (internal) concepts and concepts from other vocabularies (external). External concepts are reused as-is in the ePO Ontology.

  • Conventions used for both internal and external concepts (stored as tags in the CM):

    • skos:historyNote: Contains the date that the concept was approved by the Working Group as a rdf:plainLiteral datatype using the following notation: “WG Approval DD/MM/YYYY”. For example, “WG Approval 28/04/20.”

    • skos:editorialNote: Contains the business term used by a specific data model that can be mapped to the concept. Its value is a rdf:plainLiteral datatype using the following notation: “Used by X BT/BG-Y, Z BT/BG-H..” where X and Z are the data models, and Y and H the number of the Business Term (BT) or Business Group (BG) that this concept represents. For example, “Used by eForms BT-724".

    • dct:source: Contains the reference from where this definition was taken.

  • Conventions used for internal concepts:

    • For each internal concept a definition given to the concept by the WG is filled in the ‘Notes’ section of the Conceptual Model. As seen below. Model2owl will transfer the text contained in the ‘Notes section’ to skos:definition.

  • Conventions used for external concepts.

    • For each external concept, the ‘Notes’ section of the Conceptual Model contains a hyperlink to a human-readable website that provides the original definition of the concept. Note that the external concepts are not exported in the formal SDS artefacts (OWL and SHACL files), but can be seen in the generated Conceptual Model visual documentation” business object (html pages of the conceptual model).

  • Code lists.

    • Even though code lists are visible in the Conceptual model as enumerations, they are not exported to the OWL files since they are only recommendations of the ePO. However, they can be included or not in the SHACL files for validation purposes.

Conceptual Model version management

In this section, we describe the problem of managing the different versions of the binary EA Conceptual Model QEAX file and present the current solution. We also briefly mention two alternative ways of versioning the Conceptual Model.

  • Problem definition - the CM files QEAPX are binary and cannot be versioned properly in GitHub.

  • Current approach: pure Git – This method forces us to use Trunk based development, which requires:

    • Code freeze e.g., no new code changes or updates to the CM file are allowed to be introduced into the project, during release periods.

    • Maintaining different EA files in different branches

    • Requires re-implementation of the work done in parallel during the code-freeze. To mitigate this, good documentation practices are followed such as GitHub issues updates, and keep meeting minutes, so that it is easy to re-implement any release changes in the trunk after the lift of the code freeze. For more information on the current GitHub approach, consult section 5.

  • There are other approaches to CM versioning listed below. Both of these approaches seem problematic when compared to the current approach. It would be a good idea to further investigate these approaches or any others as future work.

    • The creators of the Enterprise Architect software have their own cloud-based infrastructure, which allows having servers as repositories of EA files with version control. However, such solutions are expensive, and experimental.

    • EA 17 also uses a new feature called package control. It uses an auxiliary XML file for each module that is saved next to the EA .qeax file. When an update to the .qeax file is made, the user can use package control to see the changes made, and either keep them, or discard them. This feature is, however, experimental and needs to be tested. Moreover, even a small change such as a slight move of an element in a diagram or the resizing of an element, will trigger a change, making keeping track of what has changed difficult.
